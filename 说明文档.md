# 说明文档

## 一、项目规划

### 1.1 项目背景

本项目基于 NewAPI（New API）进行二次开发与本地部署准备。当前代码基线来自活跃维护版本 `QuantumNous/new-api`，并已切换到用户自有 fork 进行后续开发。

### 1.2 仓库关系说明

- **原始创始仓库**：`Calcium-Ion/new-api`
- **当前活跃维护仓库**：`QuantumNous/new-api`
- **用户开发仓库**：`lee-134/new-api`
- **远程策略**：`origin` 指向 `lee-134/new-api`，`upstream` 指向 `QuantumNous/new-api`
- **本项目基线**：以后续开发均以 `QuantumNous/new-api` 的 `main` 分支为上游基线，并向 `lee-134/new-api` 推送开发提交。

### 1.3 开发约束

- **文档优先**：每次进入项目先阅读本文件，再开展后续工作。
- **进度闭环**：每完成一项任务，必须同步更新本文件中的进度记录。
- **代码规范**：后续新增或修改代码时，所有函数必须补充函数级注释。
- **运行约束**：不配置自动运行、自动部署、定时启动类能力，所有启动动作保持人工确认。
- **环境说明**：规范建议使用 Mac 开发；当前工作区实际运行环境为 Windows PowerShell，仅用于仓库同步与准备工作。

### 2.4 前端结构学习结论

当前项目前端实际由两套实现组成：

- **`web/default`**：新前端，建议作为后续二开的主入口；
- **`web/classic`**：经典前端，主要用于兼容旧页面与旧交互实现。

`web/default` 的主要特征如下：

- **技术栈**：`React 19` + `TypeScript` + `Rsbuild` + `Tailwind CSS 4` + `Base UI` + `TanStack Router` + `TanStack Query` + `Zustand`；
- **入口文件**：`web/default/src/main.tsx`；
- **路由组织**：采用 `src/routes/` 文件路由，路由树由 `routeTree.gen.ts` 自动生成；
- **认证方式**：`routes/_authenticated/route.tsx` 在路由进入前校验本地用户缓存与 `getSelf()` 会话有效性；
- **布局特点**：侧边栏使用 drill-in 视图切换模式，核心骨架位于 `components/layout/components/app-sidebar.tsx`、`top-nav.tsx`、`public-navigation.tsx`；
- **适合修改内容**：新页面、新交互、新主题、新表格体验、后续主要功能演进。

`web/classic` 的主要特征如下：

- **技术栈**：`React 19` + `JSX` + `Rsbuild` + `Semi UI` + `react-router-dom`；
- **入口文件**：`web/classic/src/index.jsx`；
- **路由组织**：所有页面路由集中在 `web/classic/src/App.jsx` 中维护；
- **状态管理**：主要基于 `StatusContext`、`UserContext` 等 React Context；
- **权限控制**：`helpers/auth.jsx` 中通过 `PrivateRoute`、`AdminRoute` 按 `localStorage` 用户信息做拦截；
- **布局特点**：`components/layout/PageLayout.jsx` 负责头部、侧边栏、内容区与页脚整体编排；
- **适合修改内容**：旧页面兼容、小范围修补、对照旧逻辑查找历史实现。

### 2.5 前端二开建议顺序

后续如果要正式改前端，建议按以下顺序阅读：

1. 先看 `web/default/src/main.tsx` 与 `web/default/src/routes/__root.tsx`，理解应用启动与根路由；
2. 再看 `web/default/src/routes/_authenticated/route.tsx`，理解登录态与受保护页面；
3. 然后按具体功能进入 `web/default/src/routes/` 和对应 `features/`、`components/`；
4. 只有在需要兼容旧控制台逻辑时，再回看 `web/classic/src/App.jsx`、`PageLayout.jsx` 与相关页面组件。

### 2.6 `antui` 绑定 `classic` 的实施策略

本轮没有直接把 `classic` 的组件源码硬搬到 `antui` 中，而是先采用“页面绑定层”方案落地，原因如下：

- **依赖差异明显**：`classic` 依赖 `Semi UI`、`react-router-dom` 等；`antui` 依赖 `Ant Design Pro`、`Umi`，两套前端基础设施并不一致；
- **直接源码迁移风险高**：若直接把 `classic` 组件导入 `antui`，会同时引入样式、路由、状态管理与依赖缺失问题；
- **先绑定后重写更稳**：先让 `antui` 页面承接 `classic` 的真实业务能力，再按页面逐步做原生改写。

当前采用的具体做法如下：

1. 在 `web/antui/src/components/newapi/classic-route-frame.tsx` 新增通用绑定容器；
2. 绑定容器通过 `/api/status` 获取后端 `server_address`，拼接 `classic` 页面地址；
3. 通过 `iframe` 在 `antui` 页面内承载对应 `classic` 路由；
4. 同步调整 `antui` 的中文菜单文案，使已绑定页面可直接按业务名称识别。
